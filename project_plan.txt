A few things worth highlighting about how this plan is structured:
The first two weeks are entirely infrastructure — no application code. This is the professional way to do it. Most people jump straight to writing features and then spend week 7 trying to figure out Terraform while also debugging their app. Setting up EKS, Aurora, ElastiCache, CI/CD, and ArgoCD first means every week after that is just git push to deploy.
Phase 3 (the feed engine) is where you should expect to spend the most debugging time. The fan-out worker sounds simple but the edge cases are tricky — what happens if a SQS message is processed twice? What if Redis is full? What if the follower list DynamoDB query takes 3 pages? Budget extra time here.
Week 10 is not optional. The demo video and the blog post are what get this project noticed. A GitHub repo with no README and no demo is invisible. Engineers have been hired specifically because a recruiter found their blog post explaining the Twitter fan-out problem.

Phase 3 — Feed engine & notifications (weeks 6–7)

The hardest and most impressive part. Build the hybrid fan-out feed system and the real-time notification service. This is the engineering that separates this project from a basic CRUD app.

Weeks 6–7 Hardest phase Fan-out + real-time
Week 6

Hybrid fan-out feed engine

Design Redis data structure: each user's feed is a sorted set — key is feed:{userId}, score is post timestamp (Unix ms), value is postId. Capped at 800 items max per user
When a post is created, Post service publishes a PostCreated event to SNS. SNS fans out to two SQS queues: FanoutQueue (for regular users) and AnalyticsQueue
Build Fan-out Worker as a separate EKS deployment. It consumes FanoutQueue messages. For each post: check author follower count. If under 10k followers, fetch all follower IDs from DynamoDB (paginated GSI query) and ZADD the postId to each follower's Redis feed sorted set
For celebrity users (10k+ followers): skip fan-out. Store a "celebrity follows" list per user in Redis. At feed read time, merge celebrity posts from a shared celebrity:{userId} sorted set into the user's personal feed
Build Feed service GET /feed endpoint: ZREVRANGE feed:{userId} 0 19 from Redis to get top 20 postIds by score. Batch-fetch post details from Post service. Merge in celebrity posts. Return hydrated feed with cursor for pagination
Handle feed gaps: if Redis feed has fewer than 5 items (new user or cache miss), fall back to a direct Aurora query joining follows and posts ordered by created_at. Backfill Redis async
HPA on Fan-out Worker deployment based on SQS queue depth (custom metric via KEDA or CloudWatch metric adapter). Scales 1→20 pods under viral load

Week 7

Notification service + search

Build Notification service. Subscribes to SNS topics: PostLiked, PostCommented, UserFollowed, PostReposted. On each event, write a notification row to DynamoDB with userId, type, actorId, postId, read=false, createdAt
Unread notification count: INCR notif:unread:{userId} in Redis on each new notification. Reset to 0 on GET /notifications (mark all read)
API Gateway WebSocket API for real-time in-app notifications. On connect, Lambda stores connectionId → userId mapping in DynamoDB. On new notification, Notification service calls API Gateway Management API to push to live connection
Email notifications via SES: first-time follower, weekly digest. Use SES templates. Rate-limit emails per user to avoid spam (Redis counter: email:{userId}:{date})
Build Search service using OpenSearch (or Aurora full-text search for simpler setup). Index users (username, bio) and posts (content, hashtags). Endpoints: GET /search/users?q=, GET /search/posts?q=, GET /search/hashtags?q=
